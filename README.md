## 💡 全局运行机制说明

在运行以上三个脚本时，系统具有以下自动保护机制：
1. **防覆盖机制（并存输出）**：为了防止同名文件输出时被覆盖，系统每次都会检查输出文件夹。如果已存在同名文件，会自动以 `(1)`、`(2)` 的形式追加序号生成新文件（例如 `测试__水印(1).png`）。**注意**：如果您不想重复生成大量文件，建议在每次批量处理前清理一下输出文件夹。
2. **异常文件跳过**：如果输入文件夹中混入了 0 字节的空文件、损坏的图片或非图片格式文件，程序会自动跳过它们并记录在日志中，不会因此报错中断。

## 1. filling_in.py — 嵌入水印

**你要做什么：**

1. 把照片丢进 `input/filling_in/covers/`（支持 png / jpg / jpeg / tif / tiff / bmp / webp）
2. 把水印/签名丢进 `input/filling_in/watermarks/`
3. 运行

```bash
python filling_in.py
```

**输出：** `output/filling_in/` 里就是含水印的图，肉眼看不出区别，你公开发布这些。

> **💡 防二次篡改机制：**
> 系统内置了特征检测。如果你放入的原图**已经包含过盲水印**，程序会精准识别并**拒绝二次打入水印**（直接跳过该图），防止他人用你的含水印图强行覆盖洗稿。

> 水印可以是任何黑白分明的 PNG——手写签名、Logo、二维码都行。

---

## 2. comparative_proof.py — 取证对比

**你要做什么：**

1. 把原图丢进 `input/comparative_proof/covers/`
2. 把含水印图丢进 `input/comparative_proof/watermarked/`
3. 把原始水印丢进 `input/comparative_proof/watermarks/`
4. 运行

```bash
python comparative_proof.py
```

> **💡 匹配规则说明：**
> 程序会自动根据**文件名**来匹配“原图”和“含水印图”。
> 例如：含水印图叫 `Cake__分形水印.png`，程序会自动在 `covers/` 文件夹里寻找名字包含 `Cake` 的原图（如 `Cake.jpg`）进行对比。
> 如果找不到名字匹配的原图，程序会**默认使用 `covers/` 文件夹里的第一张图片**作为原图。为了保证对比结果的准确性，建议保持原图和含水印图的文件名前缀一致。

> **💡 关于原图的宽容度：**
> 即使你放入的“原图”本身已经带有水印，程序依然会将其作为基准原图进行对比而不会报错。因为取证的核心是提取“含水印图”中的特征并与“原始水印”进行比对。

**输出：** `output/comparative_proof/` 里是完整证据链对比图。

**图上的判定：**
- **取证成功** — NC > 0.70，水印匹配，可证明图片出自你
- **不匹配** — NC < 0.30（≈随机噪声），该图未嵌入此水印

---

## 3. verify.py — 盲验证

> 这是你最常用的：网上看到一张图，怀疑是你的，直接验证。

**你要做什么：**

1. 把待验证的图片丢进 `input/verify/images/`
2. 把你的原始水印丢进 `input/verify/watermarks/`
3. 运行

```bash
python verify.py
```

**输出：** `output/verify/` 里是验证结果图。

**图上的判定：**
- **验证成功** — NC > 0.70，该图片包含你的水印
- **未匹配** — NC < 0.30（≈随机噪声），不含此水印

> 不需要原图，只靠待验证图片本身就能盲提取水印。

## 判定逻辑

水印是 64×64 = 4096 个像素。每个像素在嵌入时被编码为 DCT 系数的大小关系。

提取时逐像素比对，计算 **NC（零均值归一化交叉相关，ZNCC）**：
*(注：ZNCC 算法能有效消除图像亮度的影响，使得随机噪声的匹配度严格趋近于 0，完全匹配为 1)*

| NC 值 | 含义 | 判定 |
|-------|------|------|
| > 0.70 | 提取出的水印与原水印高度一致 | 包含你的水印 |
| < 0.30 | 提取出的图案接近纯随机噪声，或为不相干水印 | 不含此水印 |
| 0.30~0.70 | 介于噪声与匹配之间 | 可能有微量残留 |

## 四：水印原理

水印不会画在图上——它修改的是图片 Y 通道（亮度）的数学结构：

1. 把图片缩放到 512×512，分成 64×64 = 4096 个 8×8 小块
2. 每个小块做 DCT 变换，取出两个中频系数 D(4,3) 和 D(3,4)
3. 根据水印签名上对应位置的像素（黑/白），调整这两个系数谁大谁小
4. 逆变换回去，肉眼完全察觉不到

提取时反过来：对图片做同样的 DCT 变换，比较每个小块系数的大小关系，盲恢复出 64×64 的水印图案。不需要原图。

4096 个像素对的系数关系同时恰好匹配的概率为 (1/2)^4096 ≈ 10^-1233。

## 五：系统日志 (System Log)

为了方便排查问题，系统在运行过程中会自动在项目根目录下生成一个名为 `system.log` 的日志文件。

* **日志内容**：记录了程序的启动/结束时间、成功处理的图片信息、被跳过的异常文件警告，以及底层的报错堆栈信息。
* **滚动机制**：日志文件最大为 1MB，超过后会自动备份（最多保留 3 个历史备份），不会无限占用硬盘空间。
* **排错指南**：如果程序运行出现异常，或者某些图片没有成功输出，请打开 `system.log` 查看详细的错误原因。如果需要向开发者反馈问题，请附上该日志文件。

## 六：常见问题与系统局限性 (FAQ)

**Q: 如果有人从网上下载了我没有打水印的原图，然后打上了他自己的水印，我该怎么证明图片是我的？**

**A:** 这是一个经典的“盗贼抢注”问题（先发制人困境）。从纯技术的角度来说，盲水印系统本质上是一把**“隐形的锁”**。如果你自己不给自行车上锁（不打水印）就扔在街上，小偷推走后挂上了他自己的锁，系统检验时自然只能得出“这把锁是小偷的”的结论。

**水印技术是“事前防御”工具，而不是“事后追溯”工具。** 如果图片本身是纯净的，代码不可能凭空变出原作者的版权信息。

但在现实的版权纠纷中，原作者可以通过以下方式进行**降维打击**来证明版权：
1. **时间戳（Prior Art）**：你在微博、小红书、Pixiv 或个人网站上发布这张图片的时间，绝对早于小偷打完水印后发布的时间。互联网是有记忆的，发布时间就是铁证。
2. **源文件（终极杀招）**：原作者手里有相机的 **RAW 格式原片**，或者画师手里有带几百个图层的 **PSD 工程文件**。而小偷从网上扒下来的，只是一张被压缩过的、扁平的 JPG 图片。只要亮出工程文件，小偷瞬间哑口无言。
3. **分辨率与画质**：原作者手里有 4K 甚至 8K 的超高清无损原图。小偷扒下来的往往是平台压缩过的 1080P 甚至更糊的图。小偷就算把糊图打上水印，也变不出高清细节。

**附：关于终端乱码的说明**
> 在 Windows 终端（PowerShell/CMD）运行脚本时，可能会看到类似 `鎵惧埌 5 寮犲師鍥` 的乱码。
> * **原因**：脚本中为了强制支持跨平台 UTF-8 写入，设置了 `sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8")`。而 Windows 终端默认使用的是 GBK 编码（代码页 936）。
> * **影响**：这**仅仅是终端显示问题**，完全不影响程序的实际运行逻辑，也不影响生成的图片、JSON 文件和最终的证据对比图。如果想在终端看到正常中文，可以在 PowerShell 中运行 `chcp 65001` 切换到 UTF-8 编码后再执行脚本。
